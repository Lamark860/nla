## Phase B — выровнять frontend под новую шкалу рейтингов

**Зачем:** После v0.11.0 бекенд считает рейтинги правильно, но фронт не знает про новую шкалу. Видны два конкретных побочных эффекта:

1. `composables/useFormat.ts::ratingChipStyle()` сравнивает префиксы lowercase. На Moody's `Baa1` (нижний инвестиционный, ≈BBB+) даёт **красный** цвет (потому что начинается на `b`). Должен быть жёлтый.
2. `pages/bonds/monthly.vue:96-102` и `pages/bonds/by-issuer.vue:98-102` фильтруют по `score` 1-10. Из-за того что новая шкала через `LegacyScore10` даёт legacy-совместимые числа, фильтры работают, **но**:
   - "BB" и "B" вообще не имеют категорий в фильтре — все попадают в "BBB" бакет
   - Фильтр не различает AA+/AA/AA- между собой

**Что делать:**

- Перенести `NormalizeRating(text)` на TypeScript как pure function в `useFormat.ts` (или отдельный `useRating.ts`)
- `ratingChipStyle()` принимает `text` → внутри вызывает нормализатор → красит по `tier` (AAA/AA/A/BBB/BB/B/CCC/CC/C/D)
- Расширить фильтры в `monthly.vue` / `by-issuer.vue` — фильтровать по `score_ord` или по `tier`, а не по 1-10 score. Добавить категории «BB» и «B/CCC» отдельно
- (опционально) сделать `aiRatingStyle` тоже более тонким по +/-
- Тесты на TS-нормализатор не критичны, но желательны (vitest, дублирующие табличные кейсы)

**Объём:** ~3 часа

## Phase C — события эмитента (issuer events)

**Цель:** в табе на странице бумаги показывать ленту реальных корпоративных событий по эмитенту (отчёты, существенные факты, дивиденды, собрания) + подмешивать в AI-анализ (через отдельного AI-агента-аналитика новостей).

### Источник данных — нерешённая развилка

Изначально планировался **e-disclosure.ru RSS** — оказалось это **тупик**:
- У e-disclosure нет публичного RSS
- Сайт защищён Servicepipe (anti-bot JS challenge), `net/http` получает 403
- Нужен headless Chrome с обходом fingerprint — слишком тяжёлая инфра для pet-project

Также проверены и не подходят:
- MOEX ISS `/securities/{secid}/events.json` — алиас description, не реальные события
- MOEX ISS `description` — нет ИНН, нет событий
- MOEX `/iss/disclosure/...` — 404, такого endpoint нет

**Три реалистичных кандидата (выбор перед стартом Phase C):**

#### Кандидат 1 — `disclosure.skrin.ru` / `disclosure.1prime.ru`

Альтернативные авторизованные раскрытия (мандат тот же что у e-disclosure). **Не проверены на anti-bot** — следующий шаг разведки: один `wget`-запрос на каждый и посмотреть отдают ли HTML без challenge.

- **Если HTML отдаёт чисто:** это _именно тот источник_ который мы изначально хотели (сущфакты, отчётность, собрания), пишем HTML-парсер, идём по плану. Объём: 0.5 дня разведка + 1-2 дня интеграция.
- **Если защищены:** выбрасываем, идём на кандидат 2 или 3.

#### Кандидат 2 — Tinkoff Invest API

REST/gRPC, бесплатный с токеном (генерится в личном кабинете Тинькофф).

- **Что даёт:** структурированные данные по эмитенту/бумаге — купоны, дивиденды, корпоративные действия (split/merger), бренд+описание, фундаментальные показатели, иногда новости
- **Чего не даёт:** raw "сущфакты" в формате e-disclosure
- Тем не менее покрывает большую часть сигналов которые из сущфактов извлекаются (дефолт→корп. действие, изменение структуры → split, и т.д.)
- Хороший выбор если важны **структурированные данные** для оценки риска ВДО, а не текстовые новости
- Объём интеграции: ~2 дня

#### Кандидат 3 — Smart-lab RSS по тикеру

Простой XML-фид по тегу: `https://smart-lab.ru/q/{TICKER}/news/rss/` (URL надо подтвердить).

- **Плюсы:** нет anti-bot, простой парсер
- **Минусы:** не сущфакты, а пользовательские посты + редакционные новости. Тикер ≠ эмитент (один эмитент = много тикеров — нужна мапа). Сигнал шумный
- Подходит как «лучше чем ничего», для крупных имён работает приемлемо
- Объём: ~0.5 дня

### План Phase C (когда источник выбран)

Архитектура одинаковая независимо от источника:

1. **Расширить `bond_issuers`:**
   - `inn string` — ИНН эмитента (источник: где взять — открытый вопрос; не из MOEX ISS)
   - `external_ids map[string]string` — словарь `{"edisclosure":"123","tinkoff":"BBG..."}` для маппинга в каждый источник
2. **Новая Mongo-коллекция `issuer_events`:**
   ```
   { _id, emitter_id, source ("edisclosure"|"tinkoff"|"smartlab"|...),
     event_type ("material_fact"|"report"|"dividend"|"meeting"|"news"),
     title, summary, url, published_at, raw }
   ```
   Уникальный индекс `{source, url}` для дедупа.
3. **HTTP endpoint** (по решению пользователя — не воркер, а ручной trigger):
   - `POST /api/v1/issuer-events/sync` body `{emitter_ids: [N, ...]}` — синхронный, блокирующий, возвращает `{synced: N, errors: [...]}`
4. **Новый клиент** `internal/client/{source}/client.go`
5. **Сервис** `internal/service/event.go` — `Sync(emitterIDs)`, `GetByEmitter(emitterID, limit)`
6. **Frontend:** новый таб `BondEventsTab.vue` или секция в `BondDetailsTab`. Две колонки: «Прошедшие» и «Ожидаемые» (если источник даёт upcoming)
7. **Тесты:** парсер (RSS/HTML/JSON в зависимости от источника) — табличные

## Phase D — AI-агент аналитика новостей

**Идея пользователя:** не подмешивать сырые события в основной `bond_analyst` промпт, а сделать отдельного агента, который сначала их проанализирует и выдаст структурированное резюме рисков. Резюме передаётся в основной анализ как доп. секция.

**План:**

1. Новый промпт `data/prompts/news_analyst.txt` — фокусированный на:
   - Выявление негативных сигналов (просрочки, реструктуризации, смена контролирующего лица, неисполнение обязательств)
   - Тревожные триггеры для ВДО (downgrade, отзыв рейтинга, технические дефолты)
   - Позитивные сигналы (улучшение метрик, IPO, рефинансирование)
2. Новый сервис-метод `service.AnalyzeIssuerNews(ctx, emitterID, eventsLimit=10)` → `string` (краткое резюме)
3. Кэшировать результат в Mongo на срок последнего события (если новых событий не появилось — отдаём кэш)
4. Хук в `analysis.go`: перед формированием prompt'а основного `bond_analyst` — вызвать `AnalyzeIssuerNews`, добавить секцию "КОНТЕКСТ СОБЫТИЙ" в системный промпт

Зависит от Phase C — без событий в БД нечего анализировать.

## Открытые вопросы / задолженности

- **Как получать ИНН эмитента?** MOEX disclosure его не отдаёт. Источники: вручную через Контур/СПАРК/checko.ru (по названию), или из самих раскрытий (skrin/1prime, если получится). Если идём в Tinkoff API — там в `Asset` есть `Brand.companyInn` для российских компаний
- **Тесты на mongo-зависимый код по-прежнему отсутствуют** (`IssuerRepo`, `RatingRepo`, `EventRepo` когда появится). Это известная дыра, не блокер
- **Сравнение бумаг (compare)** и **пресеты-фильтры** (`«Стабильно надолго»`, `«ВДО на флип 6 мес»`) — обсуждались в более ранней сессии, не приоритет. Остаются в backlog
